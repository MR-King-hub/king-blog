# 大模型的输入是工程拼出来的：Embedding、RAG 与其它常用手法

> 很多人第一次听到 embedding / RAG，会以为模型被「再训练」了。其实绝大多数产品里，改的是**这一次请求的输入**，不是模型权重。本文把常见输入处理手法放在一张图里讲清楚。

---

## 先立住一个结论

和 Chat 大模型打交道时，真正进模型的通常是一段结构化消息，例如：

```text
system：你是谁、要遵守什么
user / history：对话历史（可能被裁剪或摘要）
（可选）retrieved docs：检索来的资料
（可选）tool results：工具刚返回的结果
user：当前问题
```

**Embedding、RAG、Skill、Memory、Prompt 工程……多数都在回答同一件事：这段输入怎么拼、拼什么、别超长、别跑偏。**

| 手段 | 改权重？ | 主要干什么 |
|------|---------|------------|
| 预训练 / 后训练（SFT、RLHF…） | **是** | 改模型本身 |
| Embedding 检索 / RAG / Prompt / Memory… | **否** | 工程上组装**本次输入** |

下文说的全部属于右边这一列。

---

## Embedding：语义检索的尺子

**Embedding** = 把文本编成一串数字向量，让「意思近的」在空间里靠近。

它**本身不是**改 Chat 模型输入的步骤，而是一种**检索方式**（还可以做聚类、去重、推荐）。常见用法：

1. 离线：文档切块 → 每块算 embedding → 存进向量库（向量 + 原文）  
2. 在线：问题也算 embedding → 找最近邻 → 取出原文块  

注意：向量只用来**找哪几段文字**；Chat 模型读到的仍然是**文字**，不是那串浮点数。

和关键词搜索（BM25 等）比：embedding 更擅长安词、换说法；专有名词、精确编号有时反而关键词更稳。线上常做 **hybrid**（两路一起再融合）。

---

## RAG：把检索结果贴进本次 Prompt

**RAG（Retrieval-Augmented Generation）** 是一种流水线，不是另一种模型：

```text
问题 → 检索（常用 embedding，也可用关键词）→ 把相关原文塞进 prompt → LLM 生成
```

所以：

- **Embedding ≠ RAG**：embedding 是尺子；RAG 是「先找资料再生成」整条链  
- **RAG ≠ 后训练**：资料在外库和本轮上下文里，不进 Chat 模型参数  
- 像开卷考试：每次翻书架，考完合上

**代价：** 检索错了就答偏；上下文被资料占满；延迟和费用上升。资料更新友好——改库即可，不必重训模型。

## 把输入做漂亮的其它常用技术

### 1. System prompt / 指令层

告诉模型角色、边界、输出格式。「你是客服，不要编造政策」这类，全靠写进输入，不靠改权重。

产品里的 **Skill / Rules / CODEBUDDY.md**，本质同类：**固定规程注入输入**，让 Agent 按你的规矩干活。

### 2. Few-shot（少样本示例）

在 prompt 里贴 1～几条「输入→期望输出」示范，引导格式与风格。轻量、立刻生效；占 token，示例要挑准。

### 3. 对话历史的裁剪与摘要

多轮对话会超长。常见工程：

- **截断**：只留最近 N 轮  
- **摘要**：旧对话压成一段 summary 再塞回去  
- **滑动窗口 + 关键事实外提**（用户偏好、已确认的约束）

目标都是：有限上下文窗口里，尽量保留「对下一步有用」的信息。

### 4. Chunking（切块）与 Rerank（精排）

检索质量经常死在切块：太大噪声多，太碎缺上下文。  
召回 Top-K 之后，再用更强的 **reranker** 重排，留下真正相关的几段再进 prompt——省 token、减干扰。

### 5. Tool calling（工具调用）

Agent 跑工具时，输入会多一类消息：模型提议调什么工具 → 程序执行 → **tool result 写回 messages** → 再让模型继续。

这也是在改输入：下一轮模型看到的是「带工具结果的对话」，不是魔法记忆。

### 6. Memory（记忆）

| 类型 | 典型做法 |
|------|----------|
| 短程 | 当前 session 的 history |
| 长程 | 外存笔记；用时再检索（常又走到 embedding） |
| 工作记忆 | Agent 自己维护的 todo / scratchpad，逐步写进上下文 |

Memory 产品形态很多，底层几乎总是：**外存 + 择时注入输入**。

### 7. Structured output / 格式约束

要求 JSON、表格、固定字段：靠指令 + 示例 +（可选）受约束解码 / schema。仍然是输入侧（有时加输出侧约束）工程，不是另训一个模型。

### 8. Prompt caching（提示缓存）

超长、稳定的前缀（大资料、大 system）在部分服务商可缓存，降延迟和费用。不改变「拼输入」的语义，只优化重复前缀的计费与速度。

## 一张总图：请求瞬间发生了什么

```text
用户问题
    │
    ├─►（可选）Memory / 向量库 / 关键词检索 ──► 选中的原文片段
    ├─► System / Skill / Few-shot / 摘要后的历史
    ├─►（可选）上一轮 Tool results
    │
    ▼
拼成 messages / prompt  ──►  Chat LLM  ──► 回答
         ▲
         └── 全是工程组装；默认不改模型权重
```

Agent 只是更勤快的组装者：多步规划、多轮工具、多次检索，**每一跳仍是在构造下一次输入**。

---

## 和「后训练」怎么划界

| 你想要… | 更合适 |
|---------|--------|
| 换口吻、听话、安全对齐、学一种稳定能力 | 后训练 / 微调（改权重，贵、慢） |
| 接上会变的私有知识、最新文档 | RAG / embedding 检索（改库与输入） |
| 固定产品规范、流程 | System / Skill 注入 |
| 多步干活、调 API | Agent + Tool，结果写回 messages |

常识：**能靠拼输入解决的，先别上后训练**；知识常变、要可审计「答依据哪段文档」时，RAG 往往更合适。

---

## 何时用、何时别用（很短）

- **只用 Prompt / Skill**：规则稳定、知识量不大  
- **上 Embedding + RAG**：文档多、要语义找、要可更新  
- **上 Agent**：需要多步、工具、分支决策  
- **上后训练**：风格/能力要「长在模型里」，且数据与成本谈得拢  

---

## 带回去的三句话

1. **Embedding** 是语义检索的尺子，不是 Chat 模型的后训练。  
2. **RAG** 是把检索到的文字**工程化地塞进本次输入**再生成。  
3. Skill、Memory、Tool、历史摘要……大多也是同一类：用工程决定大模型「这一眼看见什么」。
